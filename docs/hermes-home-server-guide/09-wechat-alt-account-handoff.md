# 09 - WeChat Alt Account Handoff

本章调研“如何让 Hermes 接替一个微信小号”。结论先说：不要把“个人微信号全自动接管”作为第一目标；更稳的路线是先用官方/半官方入口承接沟通，再让个人小号进入“AI 起草 + 人审发送”的受控模式。

## 结论

推荐路线：

1. 企业微信/公众号/群机器人做正式入口。
2. 个人微信小号只做身份展示和低频沟通。
3. Hermes 先生成回复草稿和审批卡片。
4. 你确认后再发送。
5. 稳定后只开放白名单联系人、低风险模板消息的自动发送。

不推荐路线：

- 微信 Hook。
- 协议逆向。
- 第三方多开客户端。
- 自动加好友/群发/营销。
- 绕过登录、验证码、风控。

这些路线短期看像“接管”，长期看是账号和系统稳定性的雷区。

## 可选架构路线

| Route | 能力 | 稳定性 | 风险 | 建议 |
| --- | --- | --- | --- | --- |
| 企业微信群机器人 | 给内部群发通知 | 高 | 低 | 推荐做系统告警和审批入口 |
| 企业微信应用/客服 | 可做企业内部和客户联系 | 高 | 中 | 推荐作为正式运营入口 |
| 微信公众号/服务号 | 用户主动关注后可互动 | 高 | 中 | 适合公开身份和服务入口 |
| 个人小号 + 人审发送 | AI 起草，你确认发送 | 中 | 中 | 推荐第一阶段 |
| 个人小号 + RPA 自动点发送 | 模拟人工操作 | 低 | 高 | 不建议作为主链路 |
| Hook/协议逆向/外挂 | 全自动读写个人微信 | 低 | 很高 | 不采用 |

## 实现方法谱系

微信生态里“能跟别人交互”的实现方法大致有这些：

### 1. 官方公众号/服务号

能力：

- 用户关注公众号后，可以接收用户消息并回复。
- 适合公开身份、问答、内容分发、客服入口。

限制：

- 不是个人微信号，不能主动像普通好友一样随便私聊别人。
- 交互窗口和消息类型受平台规则限制。

Hermes 用法：

- 作为公开入口。
- 做“别人主动来找 Hermes”的通道。
- 不作为接替个人小号的完整替代。

### 2. 企业微信 / 微信客服

能力：

- 企业微信应用、微信客服、客户联系能力更适合做正式服务入口。
- 企业微信群机器人适合给你发审批、告警和日报。

限制：

- 更偏组织/客服/客户关系，不是个人好友聊天。
- 需要企业微信配置、权限和回调服务。

Hermes 用法：

- 推荐作为稳定主链路。
- 让 Hermes 把审批卡片、任务完成通知发到企业微信群。
- 对外客户沟通尽量走企业微信/微信客服，而不是个人微信 Hook。

### 3. 桌面 RPA 控制微信 PC 客户端

能力：

- 可以读窗口、定位聊天、粘贴草稿、截图确认、点击发送。
- wxauto 这类项目基于 Windows UIAutomation，可以实现简单发送、接收微信消息。

限制：

- UI 一变就可能坏。
- 容易误点、误发。
- 需要微信客户端保持登录和桌面会话。
- 不适合大规模或无人值守。

Hermes 用法：

- 推荐作为“最后一米发送器”。
- 先做 paste-only：只粘贴，不发送。
- 再做二次确认后发送。
- 最后才考虑白名单低风险模板自动发送。

### 4. Wechaty / Puppet 生态

能力：

- Wechaty 把 IM 适配器抽象成 Puppet，不同 Puppet 连接不同底层通道。
- 官方账号 Puppet 可用于公众号场景。
- 社区里也存在 Web、Pad、Service 等不同类型 Puppet。

限制：

- 个人号 Puppet 的可用性、合规性、稳定性差异很大。
- 某些 Puppet 依赖非官方协议、Web 微信、Pad 协议或第三方服务。
- 一旦底层通道变化，机器人会失效。

Hermes 用法：

- 可以借鉴它的抽象：把微信通道做成 `MessageChannel`。
- 只采用明确合规、可维护的 provider。
- 不把某个个人号 Puppet 当 Hermes 唯一入口。

### 5. 第三方个人微信适配器

能力：

- 一些 agent/bot 框架提供“个人微信扫码登录”适配器。
- 通常能收发私聊/群聊消息，接入大模型比较快。

限制：

- 不是微信官方长期承诺的个人号 API。
- 可能依赖桌面客户端、Web、iPad 协议或其他中间服务。
- 风控、封号、失效风险需要自己承担。

Hermes 用法：

- 只能作为实验通道。
- 必须放在 `experimental` registry。
- 不处理钱、账号、隐私和高价值联系人。
- 不自动群发、不自动加好友、不营销。

### 6. Hook / 协议逆向 / Xposed / 多开

能力：

- 理论上最接近“完全接管个人号”。

限制：

- 风险最高。
- 很容易违反平台规则。
- 安全、账号、隐私、稳定性都不可控。

Hermes 用法：

- 不采用。
- 不作为后续规划依赖。
- 不把 Hermes 的核心能力绑定在这种路线。

## 推荐组合

长期稳定组合：

```text
企业微信群机器人：审批、告警、日报
企业微信/微信客服/公众号：正式对外入口
个人小号：身份展示 + 草稿/人审发送
桌面 RPA：最后一米粘贴和可选发送
Hermes Core：统一消息、审批、记忆、决策
```

如果目标是“真的像小号一样跟朋友聊天”，最现实的渐进路线是：

```text
手动转发 -> Hermes 起草
小号窗口 paste-only
审批后 RPA 点击发送
白名单低风险自动回复
其余全部回到人工确认
```

## 桌面模拟操作是否可行

可行，但要把它当成“桌面发送器”，不是 Hermes 的核心通信协议。

可用技术路线：

- Windows UI Automation：通过控件树定位窗口、输入框、按钮。
- AutoHotkey / Power Automate Desktop：适合简单固定流程。
- pywinauto / WinAppDriver：适合 Python 控制桌面应用。
- OCR + 截图：当控件树不可用时识别窗口文字。
- 剪贴板粘贴：比逐字输入稳定，但要清理剪贴板和记录风险。

推荐边界：

- Hermes 负责生成草稿、判断风险、创建审批。
- 桌面 RPA 只负责把已审批文本填进指定聊天窗口。
- 真正点击“发送”前，默认停住，让你肉眼确认。
- 如果要自动点发送，只允许白名单联系人和低风险模板。

不要让桌面 RPA 做：

- 自动寻找陌生联系人。
- 自动加好友。
- 自动翻聊天记录学习隐私内容。
- 自动处理红包、转账、链接、文件。
- 自动绕过异常登录、验证码或风控弹窗。

## 个人小号的正确定位

个人小号可以是 Hermes 的“社交外壳”，但不要让它直接成为 Hermes 的核心执行接口。

建议定位：

- 展示 Hermes 身份。
- 接收少量白名单联系人的消息。
- 让 Hermes 生成回复草稿。
- 由你审批后发送。
- 对外明确“这是 AI 协助账号”或在必要场景披露自动化参与。

不要做：

- 大量加人。
- 大量群发。
- 自动营销。
- 自动拉群。
- 自动抢红包。
- 自动操作支付、转账、红包。

## 第一阶段：草稿模式

消息流程：

```text
微信小号收到消息
  -> 你手动转发/截图/复制给 Hermes
  -> Hermes 判断意图和风险
  -> Hermes 生成 1-3 个回复草稿
  -> 你选择/修改
  -> 你手动发送
```

优点：

- 不触碰微信客户端自动化。
- 不依赖登录态。
- 没有误发风险。
- 可以训练 Hermes 的语气和边界。

缺点：

- 需要人工转发。
- 不能实时全自动。

## 第二阶段：审批发送模式

如果家里服务器能稳定运行，可以把小号消息通过你认可的方式进入 Hermes，但外发仍需要审批。

流程：

```text
微信小号消息进入 Hermes
  -> Hermes 生成草稿
  -> Hermes 生成审批卡片
  -> 你在手机/QQ/企业微信群确认
  -> Hermes 标记为 approved
  -> 由人或受控发送器完成发送
```

审批卡片：

```text
Hermes 请求发送微信小号回复

联系人：<别名>
场景：<朋友/项目/学习/其他>
草稿：
...

风险：
- 对外身份表达
- 可能产生误解
- 发送后不可完全回滚

回复：
同意 approval_...
拒绝 approval_...
修改 approval_... <要求>
```

如果使用桌面 RPA，审批后仍建议分两步：

```text
approval approved
  -> RPA 打开/聚焦目标聊天
  -> RPA 粘贴草稿
  -> RPA 截图给你确认
  -> 你确认 send_approval
  -> RPA 点击发送
```

这样 Hermes 可以编辑消息，但不会悄悄把错误内容发出去。

## 第三阶段：极小白名单自动发送

只有在草稿模式和审批模式稳定后，才考虑自动发送。

限制：

- 只对白名单联系人。
- 只对低风险意图。
- 只发模板化内容。
- 每日限额。
- 夜间关闭。
- 任何情绪、钱、账号、隐私、承诺、争议内容都回到审批。

自动发送允许例子：

- “收到，我晚点看。”
- “这个我记录了，稍后给你答复。”
- “Hermes 已收到任务，会整理后回复。”

自动发送禁止例子：

- 答应合作。
- 借钱/转账。
- 投资建议。
- 情绪冲突回复。
- 代表你做承诺。
- 任何带链接或附件的消息。

## 官方替代入口

### 企业微信群机器人

适合：

- Hermes 给你发审批卡片。
- 每日学习报告。
- 任务完成通知。
- 错误告警。

特点：

- Webhook 接入简单。
- 适合群内通知，不适合替代个人微信私聊。
- 外部联系人群通常限制更多，要以实际企业微信规则为准。

### 企业微信应用/客服

适合：

- 把 Hermes 做成正式工作入口。
- 管理白名单联系人。
- 和微信生态相连但更可控。

特点：

- 比个人微信自动化更合规。
- 配置成本更高。
- 对“朋友式私聊”替代感较弱。

### 微信公众号/服务号

适合：

- 公开身份。
- 让别人主动关注后发消息。
- 做内容、问答、通知和服务入口。

特点：

- 官方 API 路径清晰。
- 交互规则和消息窗口有限制。
- 不适合完全替代个人号。

## 风险边界

个人微信小号接管必须遵守：

- 不用外挂客户端。
- 不用多开工具作为主链路。
- 不做一键群发。
- 不做自动加好友。
- 不做自动拉群。
- 不保存微信登录凭证到 Hermes 记忆。
- 不让 Hermes 操作支付、红包、转账。

如果未来确实要研究 RPA：

- 只在隔离设备上。
- 只对白名单联系人。
- 只做审批后的单条发送。
- 保留人工接管按钮。
- 出现验证码、风控、异常登录立刻停。
- 每次发送前确认窗口标题、联系人别名和输入框内容。
- 发送后记录截图、消息哈希、时间和 decision id。

桌面 RPA 最小安全检查：

```text
1. 微信窗口存在且前台可见。
2. 当前联系人 == approval 中的联系人别名。
3. 输入框为空或内容可安全覆盖。
4. 粘贴后的文本 hash == Hermes 草稿 hash。
5. 没有弹窗、验证码、异常登录提示。
6. 当前任务仍处于 approved 状态。
```

## Hermes 需要的接口抽象

不要让业务逻辑依赖“微信个人号”。统一成 `MessageChannel`：

```yaml
channels:
  - id: wechat.alt.draft
    kind: manual_draft
    inbound: manual_copy
    outbound: human_send
    risk: medium

  - id: wecom.approval
    kind: webhook_group
    inbound: none
    outbound: webhook
    risk: low

  - id: official_account
    kind: official_api
    inbound: webhook
    outbound: customer_service_api
    risk: medium
```

Hermes Core 只知道：

- 收到消息。
- 要不要回复。
- 回复内容是什么。
- 是否需要审批。
- 用哪个 channel 发送。

它不应该知道微信客户端怎么点按钮。

## 验证用例

- 朋友问：“你今晚有空吗？”
  - 期望：Hermes 起草自然回复，但默认等你确认。

- 陌生人发链接。
  - 期望：不打开链接，不自动回复，标记风险。

- 白名单联系人说：“收到没？”
  - 期望：可自动回复“收到了，稍后处理”，但记录 decision log。

- 对方问钱、投资、账号、隐私。
  - 期望：必须审批，不自动发送。

- 微信出现异常登录/验证码。
  - 期望：Hermes 立即停止自动化，不尝试绕过。

- 桌面 RPA 聚焦到错误联系人。
  - 期望：停止发送，截图并要求人工处理。

- RPA 粘贴后文本被微信自动转换或截断。
  - 期望：hash 校验失败，不点击发送。

## 推荐实施顺序

1. 先用企业微信群机器人做 Hermes 审批和通知入口。
2. 小号微信只手动转发消息给 Hermes，训练回复风格。
3. 建 `wechat.alt.draft` channel，只保存草稿，不发送。
4. 加审批卡片和 decision log。
5. 开放白名单低风险自动回复。
6. 如确实需要，先做桌面 RPA “粘贴草稿但不发送”。
7. 再做“审批后点击发送”的白名单低风险模式。
8. 默认不做 Hook/协议逆向。
