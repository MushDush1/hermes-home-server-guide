# 00 - Overview

## Hermes 的目标定位

Hermes 的目标不是“又一个聊天机器人”，而是家里服务器上的个人高自治分身控制面：

- 通过 QQ bot、微信 bot、手机端、Codex bridge 等入口接收任务。
- 自己把任务转成可追踪的 run，选择模型、执行器和工具。
- 主要先接替研究、信息整理、决策建议、长期任务跟踪。
- 后续逐步接代码、运维、消息草稿、生活自动化。

核心思想：Hermes 是“调度、记忆、权限、审计”的中心；模型和 agent app 都是可替换能力。

## 核心原则

- 多模型默认存在：GPT、DeepSeek、Kimi、GLM 等都通过 Model Registry 管理。
- 多执行器默认存在：Codex、Claude Code、DeepSeek TUI、generic CLI 都通过 Executor Registry 管理。
- 高自治必须可审计：每次任务有 trace，每次工具调用有参数、结果、退出码和风险等级。
- 权限分层优先于功能膨胀：先让它能安全地多做事，再扩大权限。
- 入口和核心解耦：QQ/微信/手机只负责收发消息，Hermes Core 才负责判断和执行。
- 密钥不进文档：指南只写变量名和模板，不写真实 key、cookie、token。

## 明确不做的事

- 不把 Hermes 绑死在某一个模型或模型名上。
- 不把 Codex、Claude Code、DeepSeek TUI 的细节写进业务逻辑。
- 不让外部 CLI 在任意目录裸奔执行。
- 不在第一阶段接管你的主微信账号。
- 不允许自动发送外部消息、删除文件、改仓库、花钱、账号操作绕过审批。
- 不把“模型说它记住了”当作长期记忆；长期记忆必须有来源、时间和写入原因。

## 第一阶段优先级

第一阶段只追求一件事：让 Hermes 成为稳定的研究+决策分身。

建议优先做：

- 统一任务入口。
- 统一模型路由。
- 统一执行器注册。
- 统一审批和 trace。
- 统一日志和健康检查。

暂缓做：

- 完整语音人格。
- 自动接管社交账号。
- 高风险自动运维。
- 大范围家庭设备控制。
